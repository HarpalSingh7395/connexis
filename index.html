Universal Realtime Connection Manager

@connexis is a production-ready, framework-agnostic realtime client library designed to act as a connection manager for browser realtime transports.

Instead of wrapping WebSockets or SSE directly, `@connexis` pools connections, orchestrates multi-tab leader elections using SharedWorker and BroadcastChannel fallbacks, deduplicates identical subscriptions onto shared sockets (using Hybrid policies), and provides developer utilities like Koa-style middlewares and reactive framework hooks.

🔌

Transport Agnostic

Direct support for WebSocket, EventSource (SSE), and HTTP Polling with automatic connection failover.

👥

Multi-Tab Sharing

Elected leader tab maintains the network socket; followers delegate subscriptions via BroadcastChannel/SharedWorker.

⚖️

Policy Driven

Control transport sharing using Isolated, Shared, Hybrid, or custom policies based on topic and filter properties.

Installation

Install the core package along with any required transport implementations:

# Install Core
pnpm add @connexis/core

# Install Transports
pnpm add @connexis/transport-websocket @connexis/transport-sse @connexis/transport-polling

# Install React Bindings (optional)
pnpm add @connexis/react

Transports Configuration

Initialize clients using the transport that best fits your infrastructure:

1. WebSocket Transport

Bi-directional, low-latency, full-duplex communication:

import { createRealtimeClient } from '@connexis/core';
import { WebSocketTransport } from '@connexis/transport-websocket';

const client = createRealtimeClient({
  transport: new WebSocketTransport('wss://api.example.com/events'),
  connectionPolicy: 'isolated'
});

2. Server-Sent Events (SSE)

Unidirectional HTTP-based event stream from server to client, with an optional POST fallback configuration for publishing:

import { SSETransport } from '@connexis/transport-sse';

const client = createRealtimeClient({
  transport: new SSETransport('https://api.example.com/stream', {
    publishUrl: 'https://api.example.com/publish' // fallback POST url for publish calls
  })
});

3. HTTP Polling

Standard interval-driven polling fallback for legacy environments:

import { PollingTransport } from '@connexis/transport-polling';

const client = createRealtimeClient({
  transport: new PollingTransport('https://api.example.com/poll', {
    pollInterval: 5000 // poll every 5 seconds
  })
});

Authentication & Token Rotation

@connexis supports client-side authentication token handling across all transport layers (WebSockets, SSE, and HTTP Polling). It provides built-in mechanisms for dynamic token rotation and connection stabilization protection.

1. Dynamic Token Provider Callback

Instead of passing a static string, you can provide an asynchronous callback function that returns a Promise<string>. The client will invoke this callback before every connection or reconnection attempt. This is extremely useful for automatically renewing expired JSON Web Tokens (JWT) or retrieving fresh credentials from an auth provider. 

const transport = new WebSocketTransport('wss://api.example.com/events', {
  // Evaluated dynamically on initial connect and every retry
  authToken: async () => {
    const token = await fetchFreshJWT();
    return token;
  }
});

2. Connection Stabilization Protection

In many architectures, authentication validation happens post-handshake (e.g., after a WebSocket or SSE stream is opened). When invalid credentials are supplied, the server opens the connection and immediately closes it.

To prevent the client from spamming the server in an infinite reconnect loop, configure the stableThreshold option inside reconnectOptions. A connection must remain open for this threshold duration (in milliseconds) before the retry attempt counter is reset to 0. 

const client = createRealtimeClient({
  transport,
  reconnectOptions: {
    maxAttempts: 5,        // Stop retrying after 5 attempts
    delay: 1000,           // Retry every 1000ms
    stableThreshold: 5000  // Connection must stay alive for 5s to be stable
  }
});

Connection Policies

Control socket reuse and resource consumption across tabs using policy options:

Isolated Mode

Every client instance owns its own transport. Perfect for admin dashboards, query panels, or debug consoles where distinct data streams are required.

Shared Mode

Only one socket connection exists browser-wide. All tabs share this connection, delegating subscriptions to the single leader tab.

Hybrid Mode

Deduplicates identical subscriptions (same topic and filter) onto a shared connection. Unique filters automatically spawn dedicated connections to preserve server bandwidth.

Custom Policy Selector

You can provide a custom key selector function. This function receives the Subscription request object and returns a connection key string. All subscriptions resolving to the same key will share the same physical transport connection. 

const client = createRealtimeClient({
  transport: new WebSocketTransport('wss://api.example.com/events'),
  connectionPolicy: (sub) => {
    // Separate connections by user tier (passed in metadata)
    if (sub.metadata?.tier === 'premium') {
      return 'premium-conn';
    }
    return 'default-shared-conn';
  }
});

Multi-Tab Coordination

Passing a Coordinator instance to the client activates the synchronization layer. It attempts to load a dynamic SharedWorker first. If not supported (e.g. mobile Safari), it gracefully falls back to BroadcastChannel. 

import { Coordinator } from '@connexis/coordinator';

const client = createRealtimeClient({
  transport: new WebSocketTransport('wss://api.example.com/events'),
  connectionPolicy: 'shared',
  coordinator: new Coordinator('my_app_namespace')
});

Automatic Failover

When the leader tab is closed or refreshed, the coordination layer immediately sends a leader_exit broadcast. The remaining follower tabs instantly trigger a new leader election (determined by tab age). The promoted leader establishes the active socket and re-subscribes all delegated subscriptions seamlessly without message loss.

Middleware Pipeline

Apply async, Koa-style middlewares to intercept, inspect, modify, or reject inbound and outbound payloads: 

client.use(async (context, next) => {
  // Inbound or Outbound directions
  if (context.direction === 'outbound' && context.topic === 'orders') {
    context.payload.timestamp = Date.now(); // modify payload
  }
  
  // Rejecting a message
  if (context.direction === 'inbound' && context.payload.isSpam) {
    throw new Error('Rejected spam message'); // stops pipeline execution
  }
  
  await next(); // yield to next middleware in chain
});

Hooks Usage

The package exports four primary React hooks:

  • useRealtime(): Returns the active RealtimeClient instance from context.
  • useConnection(): Returns the current connection state and aggregated metrics. Updates when they change.
  • useSubscription(topic, optionsOrHandler, handlerOrUndefined) (aliased as useChannel): Establishes a subscription with automatic cleanup on component unmount. Avoids double-unsubscribe bugs in React Strict Mode.
  • usePublish(): Returns a callback function to easily publish payloads.
import { useConnection, useSubscription, usePublish } from '@connexis/react';

function Dashboard() {
  const { state, metrics } = useConnection();
  const publish = usePublish();

  // Subscribes on mount, unsubscribes on unmount. Handles filters.
  useSubscription('orders', { filter: { region: 'US' } }, (order) => {
    console.log('New US order:', order);
  });

  return (

Latency: {metrics.latency.toFixed(1)} ms

      
Throughput (Inbound): {metrics.throughput.inbound} msg/s

      
    

  );
}

API Reference

1. RealtimeClient Interface

class RealtimeClient<TEvents = any> extends EventEmitter {
  constructor(config: RealtimeClientConfig);

  /** Registers Koa-style middleware */
  use(middleware: Middleware): this;

  /** Gets aggregated connection state */
  readonly state: ConnectionState;

  /** Gets aggregated metrics */
  readonly metrics: Metrics;

  /** Subscribes to a topic (returns unsubscriber callback) */
  subscribe<K>(topic: K, handler: (data: any) => void): Promise<() => Promise<void>>;
  subscribe<K>(topic: K, options: SubscribeOptions, handler: (data: any) => void): Promise<() => Promise<void>>;

  /** Publishes data to a topic */
  publish<K>(topic: K, payload: any): Promise<void>;

  /** Completely destroys the client and terminates all connections */
  destroy(): Promise<void>;
}

2. Configuration Types

RealtimeClientConfig

interface RealtimeClientConfig {
  transport: Transport;
  connectionPolicy?: ConnectionPolicy; // 'shared' | 'isolated' | 'hybrid' | ((sub: Subscription) => string)
  reconnectOptions?: ReconnectOptions;
  heartbeatOptions?: HeartbeatOptions;
  coordinator?: ICoordinator;
  debug?: boolean;
}

ReconnectOptions

interface ReconnectOptions {
  maxAttempts?: number;           // Defaults to Infinity
  delay?: number | ((attempt: number) => number); // exponential or fixed delay
  timeout?: number;               // connection handshake timeout in ms
  stableThreshold?: number;       // stability window in ms. Defaults to 5000ms
}

HeartbeatOptions

interface HeartbeatOptions {
  enabled?: boolean;              // Defaults to true
  interval?: number;              // ping frequency in ms. Defaults to 30000ms
  timeout?: number;               // pong timeout in ms. Defaults to 10000ms
  message?: any;                  // custom heartbeat message payload
}

Metrics

interface Metrics {
  reconnectCount: number;         // accumulated reconnect attempts
  latency: number;                // average round-trip-time (ping-pong RTT) in ms
  uptime: number;                 // current connection uptime in ms
  transportType: string;          // 'websocket' | 'sse' | 'polling'
  connectionCount: number;        // number of active physical connections
  activeSubscriptions: number;    // number of active subscriptions
  leaderChanges: number;          // count of coordinator leader elections
  throughput: {
    inbound: number;              // incoming messages per second
    outbound: number;             // outgoing messages per second
  };
}

3. Event Subscriptions

The client inherits from an EventEmitter and fires the following events:

  • 'stateChange': Emitted when the aggregated connection state updates. Payload: { state: ConnectionState }.
  • 'metricsChange': Emitted periodically when connection stats are recalculated. Payload: Metrics.
client.on('stateChange', ({ state }) => {
  console.log(`Connection state is now: ${state}`);
});

client.on('metricsChange', (metrics) => {
  console.log(`Latency ping-pong: ${metrics.latency} ms`);
});
